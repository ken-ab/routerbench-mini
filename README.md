# RouterBench-Mini

**Cost-aware model reuse for multimodal agents.**

RouterBench-Mini is a small research-style benchmark for testing whether a lightweight router can decide when a cheap model is enough and when an agent should escalate to a stronger text or vision-language model.

Research question:

> Can selective escalation reach near-strong-model accuracy with much lower average cost?

## Why This Exists

Modern agent systems often have access to multiple foundation models: cheap text models, strong reasoning models, cheap VLMs, and stronger VLMs. Always calling the strongest model is accurate but expensive. Always calling the cheapest model is efficient but unreliable. This repo studies the middle ground: model routing under accuracy-cost constraints.

This project is intentionally compact. It is designed as a reproducible RA-application portfolio project around LLM routing, model reuse, multimodal agents, and efficient inference.

## Tasks

The benchmark is designed around three task types:

| Task type | Dataset | Evaluation |
|---|---|---|
| Text reasoning | GSM8K | final numeric exact match |
| Multimodal QA | ScienceQA image subset | multiple-choice accuracy |
| Agent tool use | BFCL-style function calling | function name + required arguments |

The included `data/mini_manifest.jsonl` is a small smoke-test manifest. Build a larger manifest with:

```bash
python scripts/build_manifest.py --out data/manifest.jsonl --gsm8k 100 --scienceqa 100 --save-images
```

For BFCL, download the relevant JSONL subset and pass it with:

```bash
python scripts/build_manifest.py --out data/manifest.jsonl --gsm8k 100 --scienceqa 100 --bfcl-file path/to/bfcl.jsonl
```

## Model Roles

The code uses model roles instead of locking the experiment to one provider:

| Role | Example model |
|---|---|
| `cheap_text` | Qwen3-8B |
| `strong_text` | Qwen3-32B / Qwen3-Max |
| `cheap_vlm` | Qwen3-VL-8B |
| `strong_vlm` | Qwen3-VL-32B / Qwen3-VL-235B |

Mock providers are included so the benchmark can run without API keys.

## Routers

| Router | Meaning |
|---|---|
| `always_cheap` | cheapest compatible model |
| `always_strong` | strongest compatible model |
| `rule_based` | fixed human rule |
| `selective_escalation` | cheap model first, verifier decides whether to escalate |
| `oracle` | post-hoc upper bound: cheapest correct model |

The main method is `selective_escalation`.

## Run

Install dependencies:

```bash
pip install -e ".[test]"
```

Run the mock experiment:

```bash
python -m routerbench_mini.cli --manifest data/mini_manifest.jsonl --models configs/models.mock.yaml --costs configs/costs.yaml --out results/mock
```

Run tests:

```bash
python -m pytest
```

## Output

The CLI writes:

- `results/mock/predictions.csv`
- `results/mock/summary.csv`

The most important metrics are accuracy, average relative cost, average latency, and escalation rate.

Example mock smoke-test output:

| Router | Accuracy | Avg Cost | Avg Latency | Escalation |
|---|---:|---:|---:|---:|
| `always_cheap` | 0.4167 | 1.3333 | 466.67 ms | 0.0000 |
| `always_strong` | 1.0000 | 9.3333 | 1400.00 ms | 0.0000 |
| `rule_based` | 0.9167 | 6.0000 | 1033.33 ms | 0.0000 |
| `selective_escalation` | 1.0000 | 6.8333 | 1216.67 ms | 0.5833 |
| `oracle` | 1.0000 | 5.6667 | 983.33 ms | 0.5833 |

This table is generated by deterministic mock providers, so it is only a smoke test. The real research table should be produced with actual Qwen/API providers on a larger manifest.

## Real API Setup

Copy the example config:

```bash
cp configs/models.qwen_api.example.yaml configs/models.qwen_api.yaml
```

Set your provider environment variables:

```bash
set QWEN_API_KEY=your_key
set QWEN_BASE_URL=https://your-openai-compatible-endpoint/v1
```

Then run:

```bash
python -m routerbench_mini.cli --models configs/models.qwen_api.yaml --manifest data/manifest.jsonl --out results/qwen
```
